### Fixed

- Fixed pi.registerTool() dynamic registration after session initialization. Tools registered in session_start and later handlers now refresh immediately, become active, and are visible to the LLM without /reload (#1720
(https://github.com/badlogic/pi-mono/issues/1720))
- Fixed session message persistence ordering by serializing AgentSession event processing, preventing toolResult entries from being written before their corresponding assistant tool-call messages when extension handlers are
asynchronous (#1717 (https://github.com/badlogic/pi-mono/issues/1717))
- Fixed spacing artifacts when custom tool renderers intentionally suppress per-call transcript output, including extra blank rows in interactive streaming and non-zero transcript footprint for empty custom renders (#1719
(https://github.com/badlogic/pi-mono/pull/1719) by @alasano (https://github.com/alasano))
- Fixed session.prompt() returning before retry completion by creating the retry promise synchronously at agent_end dispatch, which closes a race when earlier queued event handlers are async (#1726
(https://github.com/badlogic/pi-mono/pull/1726) by @pasky (https://github.com/pasky))

Question: what about security, malware, attacks etc?
Answer: For a single VPS running LLMs + mail + APIs, you can get to a reasonably secure setup with a few concrete layers.

1) Basic server hardening
On Ubuntu:

Create a non‑root user with sudo, disable password SSH login, use SSH keys only.

Change SSH port away from 22 (doesn’t add real security, but cuts noise).

Keep the system updated:

Enable unattended upgrades for security patches.

2) Network and firewall
Use UFW or nftables to only allow:

22 (SSH), 80/443 (web), mail ports (25/587/465) if you run BillionMail.

Ollama and internal services should listen on localhost only; only the reverse proxy is publicly exposed.

Optional but good: Fail2ban to block repeated SSH/login attempts.

3) Reverse proxy as security front
Put Nginx or Caddy in front of all web apps:

Terminate HTTPS (Let’s Encrypt).

Enforce HTTPS only (redirect HTTP → HTTPS).

Add basic auth / JWT / API keys for internal dashboards or agent APIs.

4) App‑level security
For your agent API and tools:

Require API keys / tokens for any endpoint that writes or books anything.

Rate‑limit by IP / key to avoid abuse.

Validate and sanitize user input (URLs, file uploads, prompts) before passing to tools.

5) Malware and intrusion
Don’t run things as root unless needed; use Docker with non‑root containers where possible.

Install a lightweight integrity/AV tool if you like (e.g. clamav + periodic scans), but the biggest wins are still patching and limiting exposed services.

Logins: never reuse passwords, always use unique strong passwords + 2FA on your VPS provider account and domain registrar.

6) Backups and recovery
Security also means being able to recover:

Have at least daily automatic snapshots or your own backup to object storage.

Regularly back up:

Ollama config & models list (you can always re‑pull models).

BillionMail config & mail data.

Agent API code (also in Git anyway).

Question: we can also code that one agent has the abillity to connect to sites like vast.ai and sets up server, rent, pays, and disconnects afetr use, all while the user only orders it in the Web UI, right? so we need to take the payment, lets say for 2 hours usage of a server with 24vramm, then the agent connects. we may need some kind of prepaid in our Vast ai account, or of any other provider. Yes, this is technically possible: you can build an agent that, after the user pays in your web UI, automatically rents a GPU instance on Vast.ai (or similar), uses it for a period, and then shuts it down again.

Answer:
1) What Vast (and others) give you
Vast’s API lets you:

Search offers for a specific GPU (e.g. 24 GB VRAM) and price range.
​

Create an instance by accepting an offer via a REST call (their create instance endpoint).
​

Control state: start/stop instances, set target state running or stopped.
​

Billing model:

You must preload credits into your Vast account before starting instances.

Usage is billed per second/hour until you stop/delete the instance.

Other providers like RunPod offer similar API‑driven GPU rental and serverless endpoints that you can start/stop or call programmatically.

So programmatic spin‑up / teardown is fully supported.

2) High‑level flow for your “GPU butler” agent
User purchase in your web UI

UI: “Run model X on a 24 GB GPU for up to 2 hours – 9.99 USD”.

You charge the user via Stripe/PayPal and record:

Max budget in dollars or hours.

Desired GPU specs (min VRAM, region, etc.).

Your backend / agent logic

Agent (Pi + tools) calls your own internal tools like:

search_gpu_offers(provider, min_vram, max_price) → calls Vast API.

create_gpu_instance(offer_id, template) → wraps Vast’s create instance API.
​

monitor_and_shutdown(instance_id, max_cost_or_time) → polls usage and stops at the limit.

You enforce guardrails in your tool implementation so the LLM can’t overspend past the prepaid budget.

Wallet / credit handling

On Vast: you keep prepaid credits topped up (either manually or via their autobilling).

On your side:

When user purchases, you create an internal “job budget”, e.g. “max 2 hours at up to 0.50 USD/h”.

The agent chooses an offer with price ≤ allowed rate and knows it must stop before 2h or before a spend cap is hit.

Runtime

Agent:

Starts the instance with a given image / Docker template.
​

Runs your workload (e.g. spin up an inference server, do training, etc.).

Tracks elapsed runtime, estimated cost, and stops the instance when:

The job finishes, or

The 2‑hour / budget limit is reached.

Then it can:

Save logs / results to your storage.

Tear down the instance (stop + delete if you want no storage charges).

User experience

In your web UI:

“Job running on GPU XYZ, est. cost XX, remaining time YY.”

Progress logs or dashboard (maybe via SSH/Jupyter proxy if needed).
​

When done: show link to outputs, invoice, etc.

3) Where Pi extensions fit
You’d expose your Vast/RunPod logic as Pi tools, not call Vast directly from the LLM:

tool:gpu_search_offers

tool:gpu_start_instance

tool:gpu_stop_instance

tool:gpu_estimate_cost

The LLM decides when to call them; your backend enforces what’s allowed:

Clamp price / GPU type / max hours inside the tool implementation.

Refuse or override unsafe parameters from the model.

So: with a prepaid balance on Vast (or equivalent) and a payments setup on your side, it’s fully realistic to have a “GPU rental agent” that the user just instructs from your web UI, and everything else (offer selection, spin‑up, runtime, teardown) is automated over the provider’s API.
